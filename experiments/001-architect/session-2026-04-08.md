## Phase 2: Debugging MCP Tool Registration

### Finding: Tool names with colons break Claude Code
Server connected but zero tools appeared. Root cause: tool names like `shell:exec` contain colons which produce invalid identifiers in Claude Code's `mcp__server__toolname` scheme. **Fix:** Renamed all tools to underscores (`shell_exec`). Filed as konf-dev/konf#4.

### Finding: MCP annotations cause silent tool drop
After the rename, STILL zero tools. Root cause: `.with_annotations()` in `tool_info_to_mcp()` serializes the MCP 2025-11-25 annotations field which Claude Code can't parse — it silently drops the entire tool list (anthropics/claude-code#25081). **Fix:** Removed annotations from MCP response.

### Finding: Empty inputSchema breaks tool registration
`echo` and `log` builtin tools had `inputSchema: {}` without `"type": "object"`. MCP spec requires this field. **Fix:** Changed to `json!({"type": "object"})`.

After all three fixes: **12 tools appeared in Claude Code.**

## Phase 3: E2E Validation

### system_introspect ✅
Called via MCP. Returned 12 tools with full metadata (name, description, input_schema, annotations).

### yaml_validate_workflow ✅
Generated `hello_world` workflow YAML. Kernel validated: `{ valid: true, workflow_id: "hello_world", node_count: 1 }`.

Also tested capability escalation: workflow requiring `nonexistent_tool` correctly rejected with `"Tool 'nonexistent_tool' is not registered"` and `"Workflow requires 'nonexistent_tool' but caller does not have a matching grant"`.

### shell_exec ✅
Wrote workflow YAML to `/workspace/config/workflows/hello_world.yaml` inside sandbox container. File appeared on host via shared volume.

### Git checkpointing ✅
`git add -A && git commit` inside sandbox. Commit `fd23a84` created.

### config_reload ✅
Called config_reload. Response: `{ reloaded: true, total_tools: 13, workflows_registered: 1 }`. The `workflow_hello_world` tool appeared in the engine.

### Finding: Hot-registered tools invisible to MCP clients
After config_reload registered `workflow_hello_world` as the 13th tool, ToolSearch still couldn't find it. The MCP tool list was cached at connection time. konf-mcp doesn't send `notifications/tools/list_changed`. **Workaround:** Restart MCP client.

### Workflow execution ✅
Started konf-backend in dev mode (`KONF_DEV_MODE=true`). Sent HTTP request to `/v1/chat`. Response:

```
event: done
data: {"output":{"message":"Hello from a workflow generated via MCP!"},"run_id":"f60abd00-..."}
```

### Finding: konf-mcp gives infra-level access
The `mcp_tool_context()` function creates a bare ToolContext with `capabilities: [tool_name]` — only the specific tool being called. No namespace, no actor identity. The HTTP backend (konf-backend) properly injects `namespace: konf:default:dev_user` in dev mode.
