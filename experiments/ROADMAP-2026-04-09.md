## The Thesis

The brain dump in experiment 002 describes a "civilization of agents" — poll-based orchestration, onion permissions, moral compass, self-healing bureaucracy, token economy, job boards, observatory.

After reading every architecture doc, every finding, every spec — **this vision is not speculative.** The primitives already exist. The gap is in completing them and composing them, not in building new abstractions.

| Brain Dump Idea | Already Exists As | What's Missing |
|---|---|---|
| Onion permission layers | `ActorRole` (6 roles × 3 levels), namespace hierarchy `konf:product:user` | Not enforced in MCP (finding 004) |
| Capability matrix | `CapabilityGrant` with parameterized bindings + `VirtualizedTool` injection | Already works — just not wired for MCP sessions |
| Redaction / context filtering | `VirtualizedTool` strips unauthorized params before tool sees input | Per-event redaction for bus (Phase F3) |
| Token economy / resource budgets | `ResourceLimits` (steps, timeout, concurrency, depth, runs per namespace) | Per-namespace cumulative quotas (tokens/day) |
| Moral compass | Design principle #7: "self-modification is validation-first" + ExecutionHooks design | Hooks not yet implemented as interceptors |
| Hiring/firing agents | `WorkflowTool` + `config_reload` + dynamic tool registration | No runtime-level dynamic process spawning |
| Observatory | `EventJournal` + `RuntimeMetrics` + `ProcessTree` + audit log schema | No UI (Appsmith planned), no OTEL export |
| Conversation bus | Nothing — this is genuinely new | Needs `tokio::sync::broadcast` bus as kernel primitive |
| The Broker | `project.yaml` trigger routing + workflow-as-tool composition | Just a product config pattern, not a new mechanism |

**The architecture was already designed for this.** The design principles say it explicitly:
- Principle #1: Rust = mechanisms, workflows = policies
- Principle #2: Three abstractions (ExecutionScope, Tools, Workflows) — "if you can't express it with these three, the abstraction is wrong"
- Principle #4: Workflows are the universal composition unit
- Principle #8: The capability lattice only attenuates, never amplifies
- Principle #9: Products are configuration, not code

The civilization IS the product layer. The mechanisms are the kernel.

### Phase F: Civilization Primitives

The brain dump's vision, translated into concrete Rust mechanisms and YAML policies. Each item traces to a brain dump idea AND an existing primitive.

#### F1: Oversight Interceptor (Brain Dump: "Moral Compass")

**What:** Pre- and post-execution hooks that intercept tool invocations. A workflow (the "oversight agent") runs before/after tool calls and can approve, reject, or modify them.

**Why it matters:** Autonomous agents (experiment 004) need guardrails that aren't prompt-based. The moral compass must be structural, not advisory.

**Maps to existing:**
- Design principle #7 (self-modification is validation-first)
- `ExecutionHooks` design already in runtime spec
- `ToolContext` already carries capability metadata

**Implementation:**
- Add `HookRegistry` to `konf-runtime` — pre_tool, post_tool, pre_workflow, post_workflow
- Hooks are workflows themselves (mechanism/policy separation)
- `oversight.yaml` is a product-level workflow that runs on every tool invocation matching a pattern
- Hook workflows get the proposed action as input, return approve/reject/modify
- Configurable in `project.yaml` under `hooks:` section

**Prerequisite:** Finding 009 fix (hooks call oversight workflows, which are workflow tools)

#### F2: Event Bus (Brain Dump: "Poll-Based Conversation Bus")

**What:** A `tokio::sync::broadcast`-based event bus in `konflux-core`. Agents subscribe to filtered event streams and react autonomously.

**Why it matters:** The push model (user → workflow → execute) handles request-response. But autonomous agents need to monitor and react: "when memory changes, update the summary." "When a workflow fails 3 times, escalate." The poll model enables this.

**Maps to existing:**
- `EventJournal` already records events but is write-only/query-only
- `RuntimeMetrics` already tracks aggregate state
- `StreamReceiver` already implements real-time event delivery per-run

**Implementation:**
- `SystemEventBus` in `konflux-core` using `tokio::sync::broadcast`
- Events: `ToolInvoked`, `WorkflowStarted/Completed/Failed`, `MemoryChanged`, `ConfigReloaded`, `CapabilityDenied`
- `EventSubscription` with filter (namespace prefix, event type, tool pattern)
- `RedactionFilter` wraps subscription with capability-based event filtering (agents only see events in their namespace)
- Trigger workflows: `project.yaml` gains `on:` section mapping event patterns to workflows

**Does NOT replace push model.** Push (request-response) and poll (event-driven) coexist. Push for user interactions, poll for autonomous monitoring.

#### F3: Dynamic Process Management (Brain Dump: "Hiring/Firing")

**What:** Runtime API to spawn and terminate agent processes dynamically — not just tool registration but full lifecycle management.

**Maps to existing:**
- `config_reload` already registers/unregisters tools dynamically
- `WorkflowTool` already wraps workflows as callable tools
- `ProcessTable` already tracks running processes
- `Runtime::cancel()` already terminates with propagation

**Implementation:**
- `Runtime::spawn_agent(config: AgentConfig) → AgentId` — creates a long-lived agent process with its own scope, polling the event bus
- `Runtime::terminate_agent(id: AgentId)` — graceful shutdown with cleanup
- `AgentConfig` is YAML: identity, capabilities, subscriptions, workflows, resource limits
- Agent processes are entries in the ProcessTable with `RunStatus::Running` and no expected completion
- Supervisory workflows can spawn/terminate agents — "hiring" and "firing" are tool calls

**Prerequisite:** F2 (agents need the event bus to poll)

#### F4: MCP Session Capabilities (Brain Dump: "Onion Layers" + Finding 004)

**What:** MCP connections get scoped capabilities based on authentication, not infra-level access.

**Maps to existing:**
- `ExecutionScope` + `CapabilityGrant` already parameterize access
- `ActorRole` enum already defines the hierarchy
- `VirtualizedTool` already enforces namespace injection
- konf-backend already does this for HTTP (dev mode injects namespace)

**Implementation:**
- MCP connection handshake includes auth token
- `mcp_tool_context()` resolves token → ActorRole → ExecutionScope
- Tool list is filtered by capabilities (agents only see tools they can call)
- Each MCP session gets a scoped `ExecutionScope`, not bare `capabilities: ["*"]`

## The Broker Pattern (Brain Dump: "Middleman AI")

This deserves special mention because it's NOT an infrastructure piece — it's a product pattern.

The Broker is a Konf product (`products/broker/`) that:
1. Receives unstructured user intent
2. Routes to specialized sub-agents (other products) via workflow-as-tool
3. Aggregates results
4. Manages cognitive load (only surfaces what matters)

**This requires no new mechanisms.** It's a `project.yaml` + workflows + prompts. The existing architecture supports it today (assuming finding 009 is fixed). The broker is the first real product that exercises the civilization model.

It should emerge naturally from Experiment 003 or 004, not be designed top-down.
