**Architecture properties proven:**
- Namespace injection (every tool gets scoped context)
- Capability attenuation (child can never exceed parent)
- Mechanism/policy separation (engine is Rust, products are YAML)
- Backend-agnostic storage (MemoryBackend trait exists, one impl: smrti)
- Deterministic control of non-deterministic models (workflows govern LLM)

**What's NOT built yet from the master plan:**
- E2: Agent-mode ai_complete — **DONE** (2026-04-09). Capability-enforced ReAct loops, dynamic tool resolution, per-node config overrides, streaming events.
- E3: products/architect as a real product — config exists but incomplete

- G3: OpenTelemetry + OpenAPI docs — not started

## 9. Key Technical Facts for Future Sessions

- YAML node field is `with:` not `input:` (only `with:` in `parser/schema.rs`)
- Workflow input stored as `state.set_output("input", input)` — templates use `{{input.x}}`
- `register_as_tool: true` + non-empty `capabilities` required for callable workflows
- Single entry node (first in YAML) — parallel via `then: [a, b]` fan-out
- MCP tool names must not contain colons (Claude Code limitation)
- MCP annotations must not be serialized (Claude Code drops all tools — anthropics/claude-code#25081)
- `RunStatus::Completed` now stores output (was returning `Value::Null`)
- `Engine::remove_tool()` added for config_reload
- MCP context gives `capabilities: ["*"]` for infra-level access (finding 004)
- Sandbox container: `konf-sandbox`, Ubuntu 24.04, UID 1000, no network, resource-limited
- LLM: ollama at `localhost:11434`, OPENAI_API_KEY=ollama
